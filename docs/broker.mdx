---
title: "Broker"
description: "A local JSONL broker protocol for token-saving agents that need bounded VRT operations without shell escape."
---

`vrt broker serve` starts a local line-oriented JSON broker over stdio. It is designed for agents and token-saving wrappers that want a compact control plane without implementing the full MCP JSON-RPC surface.

The broker exposes the same bounded VRT operations as MCP and deliberately does not expose arbitrary shell execution. Every verification run still goes through the VRT planner, runner, run lock, raw logs, and evidence ledger.

## Commands

```bash
vrt broker start
vrt broker stop
vrt broker status
vrt broker status --json
vrt broker serve
vrt queue status --json
vrt queue cancel <job-id> --json
vrt lock list --json
vrt lock show <resource-id> --json
```

`vrt broker start --json` writes repo-local broker state to `.vrt/broker/state.json`. It records the socket path, runner-pool limits, queue counters, lock counters, and enabled broker capabilities. It is intentionally repo-local and optional; if no broker is running, direct CLI verification still works.

`vrt broker status --json` returns protocol metadata, broker state, the current repository root, active run lock state, latest evidence summary, session view, allowed tools, and forbidden capabilities.

`vrt queue status --json` reads `.vrt/broker/jobs/*.json` to expose queued, running, and cancelled job counts, waiting jobs, running jobs, and runner-pool limits. `vrt queue cancel <job-id> --json` marks a still-queued job as cancelled in that repo-local ledger. `vrt lock list --json` exposes active broker locks plus lock observations from latest evidence. This gives agents a control plane for queue/lock state without giving them arbitrary shell execution.

`vrt verify --broker` and JSONL `run_verification` write job state to `.vrt/broker/jobs/<job-id>.json`. The resulting evidence includes `broker_job_id`, queue wait, lock wait, resource locks, runner pool, and singleflight metadata.

## Protocol

Each request is one JSON object on one line. Each response is one JSON object on one line.

Requests can use either `op` plus `arguments` or `method` plus `params`:

```json
{"id":"1","op":"status"}
```

```json
{"id":"2","method":"plan_verification","params":{"mode":"merge","token_profile":"rtk"}}
```

Responses keep the original `id` and return an `ok` flag:

```json
{"id":"1","ok":true,"result":{"protocol":"vrt-broker/1"}}
```

Unknown operations are structured errors:

```json
{"id":"bad","ok":false,"error":"Unknown broker operation: run_any_shell_command"}
```

## Operations

- `status`: inspect broker protocol, latest evidence, active lock, session view, and allowed tools.
- `analyze_change`: profile the current diff and risk tags.
- `plan_verification`: return the verification plan for a mode and token profile.
- `run_verification`: execute the bounded verification plan and write evidence.
- `explain_failure`: summarize the latest failed evidence, or pass `evidence_id` to target a specific record.
- `get_evidence`: return the latest evidence, or pass `evidence_id` to target a specific record.
- `escalate_verification`: compute stricter follow-up checks for residual risk.
- `get_broker_status`: return broker state, queue summary, lock summary, sessions, and latest evidence.
- `list_sessions`: list session registry and worktree sessions.
- `show_session`: show one session registry entry by `session_id`.
- `list_queue`: inspect queued/running/cancelled broker jobs and runner-pool state.
- `cancel_job`: mark a still-queued broker job cancelled by `job_id`.
- `list_locks`: inspect active and observed resource locks.
- `start_session`: inspect the current session or create a managed worktree session with `worktree_path`.
- `close_session`: mark a session registry entry closed by `session_id`.
- `session_view`: aggregate worktree sessions, latest evidence, active locks, confidence, and false-confidence counts.
- `shutdown`: ask a long-running broker process to exit after responding.

## Queue And Locks

The broker control plane distinguishes waiting from failing:

- `queued` means a verification job is waiting for ordering, runner-pool capacity, a lock, or a singleflight leader.
- `waiting_lock` means another job owns a resource such as `.next`, `prisma-client`, `port-3000`, or a browser pool.
- `singleflight` means another identical verification can provide scoped evidence when inputs match.
- `cancelled` means the job was still queued when `vrt queue cancel <job-id>` marked it cancelled; running or completed jobs are not retroactively cancelled.

Agents should not bypass VRT with a direct `build`, `test`, or `prisma generate` command while VRT reports a queued job or held resource for the same scope.

When a follower joins an identical in-flight verification, it receives its own evidence record. That record points to the leader via `continued_from` and `singleflight.shared_from_evidence_id`; it does not broaden the proof beyond the matching command/input scope.

Brokered verification acquires exclusive lock directories under `.vrt/broker/locks` before running checks that write shared resources. Examples include `.next`, `prisma-client`, `port-3000`, and `playwright-browser`. If a lock is held, VRT waits and records `lock_wait_ms` plus per-resource `waited_ms` in the evidence. Stale or unreadable lock files are cleaned before retrying; interrupted jobs are not converted into passed evidence.

## Token-saving compatibility

RTK and Headroom can call the broker with `token_profile` set to `rtk` or `headroom` on planning and run operations. The response preserves evidence ids, hashes, skipped-check residual risk, and raw log pointers so agents can summarize aggressively without losing verification proof.

```bash
printf '%s\n' \
  '{"id":"status","op":"status"}' \
  '{"id":"plan","op":"plan_verification","arguments":{"mode":"merge","token_profile":"headroom"}}' \
  '{"id":"run","op":"run_verification","arguments":{"mode":"dev","token_profile":"rtk"}}' \
  '{"id":"stop","op":"shutdown"}' \
  | vrt broker serve
```

Use MCP when the host already supports Resources, Prompts, and Tools. Use the broker when a local agent runtime only needs JSONL request/response operations.
